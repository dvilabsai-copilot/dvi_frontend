# Existing Playwright suite audit

Audit date: 2026-07-14. The active workspace is the source of truth; no production test was executed.

## Baseline and active counts

- Pre-change inventory: 52 files, 41 spec files, and 85 test-declaration lines.
- Current inventory: 69 files under `tests/e2e`, 45 spec files, and 110 test-declaration lines. The increase includes the safety harness, shared fixtures, public-route smoke tests, authenticated read-only page loads, parameterized fixture routes, hotel/driver/staff/activity/location/agent/settings workflows, page inventory, generated coverage matrix, and explicitly blocked fixed-record fixtures.
- Frontend routes: 96 declarations and 95 unique paths from `src/App.tsx`.
- Frontend pages: 295 `.tsx` files under `src/pages`; 56 are mapped to active route components by the generated page inventory.
- Frontend services: 34 files under `src/services`.
- Backend controllers: 42 files under `api.dvi.travel/src`.
- Backend endpoint decorator lines: 592.
- Static control inventory: 2,064 controls across page, component, and layout sources, including custom React/Radix buttons, links, tabs, triggers, switches, and role-based controls.
- Static API inventory: 990 entries (43 UI-reachable, 423 UI-supporting, 524 backend-contract), including direct `fetch` and Axios calls; 31 external-provider boundaries are blocked from deterministic coverage.
- Machine-readable coverage matrix: `tests/e2e/coverage/coverage-matrix.json`, regenerated by `npm run e2e:inventory` and validated by `npm run e2e:coverage`.

## Safety findings

- `production-hotel-api.spec.ts`, `production-hotel-creation.spec.ts`, and `seed-booking-test-data-prod.ts` are quarantined under `tests/e2e/quarantine`. They are excluded by default and are not runnable through the default project.
- `razorpay-local-sandbox.spec.ts`, `axisrooms-munnar-hotel-details.spec.ts`, and `hotel-pricebook-axisrooms-153.spec.ts` are under `tests/e2e/external-sandbox`; the PHP/Nest parity test is under `tests/e2e/legacy-environment`. All require explicit projects.
- Historical hard-coded credential fallbacks and legacy aliases were found in the original suite and replaced with `E2E_ADMIN_*` environment variables. The scanner now reports no blocking credential findings and never prints values.
- 40 classified files reference fixed records or IDs. These remain documented in `existing-test-classification.json` and are migration targets; they are not silently treated as E2E-owned data.
- 17 executable files contain destructive or mutating workflows; four lack cleanup evidence in the raw classification, all in excluded quarantine/external-provider history. Ordinary destructive workflows now have explicit cleanup evidence; the vehicle-registration regression remains blocked because the backend exposes no delete endpoint.
- 73 arbitrary waits occur across 14 files. They are recorded as refactor work; the new smoke harness uses semantic waits only.
- 20 skip/serial/conditional-skip findings were identified. The most important order-dependent cases are the guide live-insert/pricebook/module flows and booking scenario tests that rely on generated environment IDs.
- Oversized specs: itinerary fit-here day1 (1,599 lines), auto-preview day1 (900), guide live-insert (525), guide module (487), confirm-booking modal (477), and vehicle-only itinerary (457).
- Repository-wide frontend lint remains a documented blocker: `npm run lint` reports 1,885 findings (1,784 errors and 101 warnings), predominantly pre-existing `no-explicit-any` issues. The changed safety harness and migrated cleanup specs pass focused ESLint.

## Coverage baseline

The route inventory currently records 63 covered routes, 28 evidence-backed blockers, and 5 redirects. The page inventory records 295 page files, 56 route-mapped page components, and 45 covered reachable page components; the remaining 11 reachable components inherit explicit fixture/provider blockers. The role matrix directly covers 56 super-admin and 60 agent route entries. The control inventory records 2,064 statically discovered page/component/layout controls; focused public registration, login, vendor, hotel, driver, staff, activity, locations, agent, cities, global-settings, GST, hotel-category, vehicle-type, language, role-permission, subscription-plan, and inbuilt-amenities read-only workflows currently verify 107 controls. The API inventory records 41 verified contracts: 24 UI-reachable workflows plus 17 explicitly mapped internal list/request contracts from authenticated workflows. Nineteen UI-reachable records still lack exact workflow evidence; owning-page references are reported separately and are not treated as API coverage. One UI-reachable external-provider contract remains blocked pending an explicit deterministic provider mock.

Existing domain knowledge is retained for itinerary anchor placement, preview/confirm replay, arrival policy, hotel continuity, booking transitions, nationality handling, pricebooks, vehicle-only itineraries, and vendor/guide/vehicle workflows. The migration map is represented by the machine-readable classification file and should be extended as each large spec is split.

## False-pass and fragility findings

The baseline contains conditional interactions that can skip required work, index-based selectors, serial state, fixed IDs, and arbitrary waits. These are not reclassified as covered by this audit. New coverage must use semantic locators, direct required assertions, unique `PW_E2E_<runId>_*` records, and cleanup registered to that run ID.

## Environment and execution

Required values are documented in `.env.e2e.example`. `tests/e2e/setup/environment-preflight.mjs` rejects missing values, production hosts, unsafe database names, invalid run IDs, and unconfigured external-mock mode. The default Playwright project is smoke-only and excludes quarantine, external-sandbox, and legacy-environment directories.

## Defects found and corrected

- Vitest had no E2E exclusion, so `npm test -- --run` discovered Playwright files and failed 43 suites during collection. Adding `tests/e2e/**` to `vitest.config.ts` reduced the unit run to its four intended files and all nine tests passed.
- Login labels were not associated with inputs and the submit button exposed “Login” while tests and the workflow use “Sign in”. `src/pages/Login.tsx` now supplies input IDs/`htmlFor` labels and a stable accessible name; the login unit tests pass.
- Vendor unit expectations referenced the obsolete `/vendor/add` path and server-side search behavior, while the active application uses `/vendor/new` and client-side filtering. The test and search input were aligned with the active contract, and the vendor unit tests pass.
